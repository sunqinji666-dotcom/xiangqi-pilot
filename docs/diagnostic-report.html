一、总判断：为什么"用不了"

不是某一个功能有 bug，而是整条链路上的每一环都在叠加失败概率。一次成功的操作需要连续通过：

每个环节 90% 的成功率，串联起来就是 59%。实际上识别和点击环节的成功率远低于 90%。

二、识别模块

致命requiresHumanReview 几乎永远为 true

XiangqiVisionRecognizer.swift:55-60

confidence = mean × completeness，当 32 个子只有 2 个已知颜色时，completeness = 2/32 = 0.0625。即使 mean = 0.99，最终 confidence = 0.062，远低于 0.985 的门槛。加上 hasCompleteClassification 的冗余检查，几乎任何有 unknown 子的局面都会触发人工审核，自动模式形同虚设。

致命796 行巨型类承担 5 种不同职责

XiangqiVisionRecognizer.swift

OCR、模板匹配、占位检测、颜色估计、补丁比对全部塞在一个类里。任何一项修改都需要重新理解整个文件。不同棋盘皮肤的适配需要在十几个耦合的阈值间反复试错。

高20+ 硬编码魔法阈值，无系统性推导

散布在多个文件中的关键阈值举例：

阈值位置用途
0.30 / 0.018XiangqiVisionRecognizer:107模板匹配接受条件
0.985XiangqiVisionRecognizer:304人工审核门槛
0.62 / 0.025XiangqiVisionRecognizer:499占位检测亮度/对比度
red - max(g,b) >= 8XiangqiVisionRecognizer:690红方颜色判定
0.42 / 0.50 / 0.18GridStoneRecognizer:53-63五子棋颜色分类
0.675, 0.355, 0.285, 0.365XiangqiWizardMoveLogReader:100着法列表裁剪区域

换一套棋盘皮肤就需要重新校准这些数字，且它们之间存在微妙耦合——改一个可能影响其他所有判定。

高颜色估计过于粗糙

XiangqiVisionRecognizer.swift:673-696

红方判定用绝对差值 >= 8 + 比例 >= 1.10，偏暗的红色棋子容易失败。黑方判定 max(g,b) - r >= 4 极其宽松，任何略微偏蓝/绿的像素都判为黑方。GridStoneRecognizer 用 12×12 像素采样判定颜色，抗锯齿和压缩伪影就能搞乱结果。

高硬编码绝对路径 + 静默失败

XiangqiVisionRecognizer.swift (XQWizardTemplateLibrary)

模板库直接硬编码 /Applications/象棋巫师.app/Wrapper/XQWizard.app。如果用户安装在其他位置，模板加载返回空数组，模板匹配路径完全不工作，但没有任何错误提示。

中占位检测与 OCR 取并集导致频繁触发审核

XiangqiVisionRecognizer.swift:434-438

每次 OCR 漏识别一个子，occupiedCells 会包含这个格子但 pieces 不会——直接触发人工审核。自动模式下导致频繁暂停。

中StableFrameGate 的 120ms 稳定等待 + 帧差检测误判

BoardFrameDifferencer.swift:132-163

棋子移动动画会导致帧间差异被误判为 materialChange，重置 stableSince，门控长时间不通过。两个不同的 minimumScore 阈值（0.07 vs 0.035）没有文档说明。

中~75 行死代码 + 3 个永远返回 nil 的 stub 方法

XiangqiVisionRecognizer.swift:575-648 (inferRepeatedPieceTemplates 从未被调用)

RecognitionTransitionPolicy.swift:56-74 (decoratedLegalLine/Move/Reply 始终返回 nil，但 Runtime 中有 5 处调用)

三、点击执行

致命10+ 层验证让成功路径极窄

ClickExecutor.swift:597-700

一次点击需要连续通过：状态机检查 → 无进行中动作 → 起终点不同 → 获取帧 → epoch 检查 → 12 项前置条件（权限×2、hash、PID、windowID、geometryHash、frameSequence、contentFingerprint、fingerprint、stableFrames、imageSize、窗口权威）→ ROI 检查 → 遮挡检测 → 再检查 epoch → 延迟 260ms → 又检查 epoch → 窗口几何验证 → 又遮挡检测 → 第二次点击 → 30 秒超时验证 → 最终帧验证。

权限检查执行了 4 次，ensureStillExecuting() 调用了 5 次。大量重复检查不仅拖慢速度，还增加了某一环意外失败的概率。

致命safetyEpoch 机制让飞行中的点击必然失败

ClickExecutor.swift:185-210

任何外部 pause() 调用都会递增 safetyEpoch，导致正在执行的点击立即失败。而帧捕获循环中的 reconcileTrustedPosition() 检测到视觉变化就会触发 pause——点击本身引起的画面变化反而会中断点击。

致命verify() 帧指纹竞态

ClickExecutor.swift:745-760

要求点击后帧指纹必须与点击前不同。但如果棋子移动动画尚未完成，恰好捕获到与之前指纹相同的中间帧，验证就会失败。20fps 帧率 + 120ms 动画时间，这种竞态完全可能发生。

高错误恢复只有一种策略：暂停

PilotRuntime.swift (pause 方法)

无论权限丢失、窗口消失、帧过期、状态不匹配还是点击失败，唯一的应对都是暂停等用户手动干预。窗口轻微移动（重新获取 calibration 就能解决）和帧短暂不稳定（等几帧就好）这种常见问题也不会自动恢复。

高calibration 尺寸容差仅 0.5 点

BoardCalibration.swift (matchesSize)

macOS 在 HiDPI 和非 HiDPI 显示器之间移动窗口时，窗口尺寸可能发生亚像素变化（800.0 → 799.5），直接导致 calibration 失效抛出 windowGeometryChanged。

中窗口激活的竞态条件

ClickExecutor.swift:218-330

Accessibility 提升窗口和帧稳定检查之间，macOS 可能正在执行窗口动画。120ms 的稳定窗口要求不足以跨越完整的动画周期。

四、运行时状态管理

致命三套状态系统，两套是废的

项目中存在三套独立的状态管理：

系统状态实际情况

致命presentation.phase 在 40+ 处直接赋值，无约束

PilotRuntime.swift + PresentationModels.swift

任何地方都可以把 phase 改成任何值。可以从 .acting 直接跳到 .observing（跳过 verifying），也可以在 .previewing 时变成 .thinking。没有编译时或运行时保障来防止非法状态转换。

高双重 ObservableObject 导致数据流分叉

PilotRuntime 和 PilotPresentationModel 都是 ObservableObject，各自有大量 @Published 属性。一次业务操作可能同时修改两个对象，触发两次独立的 SwiftUI 更新周期，且更新顺序不确定。

五、架构层面

致命PilotRuntime.swift: 4,239 行的 God Object

一个文件承担了 12+ 个不同职责：设置向导、权限管理、窗口捕获轮询、棋盘标定、象棋识别调度、网格棋识别调度、引擎分析、着法执行、画面校验、异常恢复、AI 协同桥接、会话持久化。

这是所有问题的放大器——任何修改都需要在这个巨型文件中理解上下文，改一处影响全局。

高PresentationModel: 40 个 @Published + 19 个回调闭包

PresentationModels.swift (755 行)

恢复系统独占 14 个状态字段，象棋和网格棋状态混在一起。每次任何状态变化都触发所有观察者的 View 重新求值。19 个 onXxx 闭包使得数据流极难追踪。

中View 直接通过 Binding 修改 Runtime 属性

SetupWizardView.swift

用户可以在识别流程进行中改变 sideToMove，没有任何业务验证。

中synchronizePresentation 手动调用，散布在多个路径

如果某个路径忘记调用，UI 就显示过期数据。没有编译时保证，也没有运行时断言。

六、重构路线图

建议分三个阶段，每个阶段独立可交付、可测试：

Phase 1 — 打通最小可用链路（1-2 周）

目标：让"识别 → 分析 → 点击"这条路至少能跑通

  • 修复 requiresHumanReview 的逻辑 bug：移除 confidence 中 completeness 的惩罚因子，或把阈值从 0.985 降到合理水平
  • 精简 ClickExecutor 的验证链：权限检查只保留一次，移除重复的 ensureStillExecuting，合并窗口验证
  • 修复 safetyEpoch 竞态：execute() 执行期间冻结 epoch，或改用 scoped token 机制
  • calibration 容差放宽：从 0.5pt 改为 2-3pt，或加入自适应容差
  • 添加自动恢复路径：窗口轻微移动 → 自动重新获取 calibration；帧短暂不稳定 → 等待 N 帧后重试
  • 删除死代码：SessionStateMachine（或接入使用）、inferRepeatedPieceTemplates、stub 方法
Phase 2 — 识别系统重建（2-3 周）

目标：识别准确率从"基本不可用"提升到"大多数场景可用"

  • 拆分 XiangqiVisionRecognizer：分离 OccupancyDetector、SideEstimator、TemplateMatcher
  • 将魔法阈值提取为可配置的 RecognitionProfile：每个客户端/棋盘皮肤一套参数，不再硬编码
  • 改进颜色估计：用 HSV 色彩空间替代 RGB 绝对差值，增大采样区域
  • 修复 OCR 候选选择逻辑：不再选择"低置信度"候选
  • 移除硬编码路径：模板库通过用户选择或自动发现加载
  • 统一坐标系统：定义 ScreenPoint / ImagePoint / NormalizedPoint 类型，利用编译器防止混淆
Phase 3 — 架构治理（3-4 周）

目标：代码可维护、可测试、可扩展

  • 拆分 PilotRuntime：SetupCoordinator、CaptureLoop、AnalysisCoordinator、ExecutionCoordinator、RecoveryCoordinator
  • 统一状态管理：让 SessionStateMachine 成为唯一真相源，phase 变为派生属性
  • 合并双重 ObservableObject：一个 ObservableObject，分拆为子模型（RecoveryModel、GridGameModel）
  • 消除重复领域模型：View 层直接使用 XiangqiCore 类型 + 渲染扩展
  • 用协议替代 19 个闭包：定义 PilotRuntimeDelegate 协议
